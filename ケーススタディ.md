# ケーススタディの概要

ケーススタディを入れるなら、**答えを当てる**のではなく、**責任境界を考えさせる形**がよいと思います。

---

## ケース1：仕様変更

#### ケース

お客様か「この機能、やっぱり追加できますか？」  
と口頭で依頼がありました。

条件：納期は変更なしです。

#### ワーク

次の中で最初に確認することは何でしょう？

1. 開発メンバーへ作業依頼する  
2. ベンダーへすぐ連絡する  
3. 変更内容・影響範囲・責任者を確認する  
4. とりあえず了承する

ここで重要なのは正解ではなく、なぜその確認が必要なのかを話し合わせることです。

---

## ケース2：認識違い

#### ケース

リリース直前

顧客：「この機能入ってないですよね？」  
ベンダー：「依頼されていません。」  

条件：議事録を見ると曖昧な表現となっている

#### ワーク

原因は何でしょう？

- 仕様管理
- 合意形成
- コミュニケーション
- 責任境界

複数回答でも問題ないように旨を伝える。

その後、この状況を防ぐには何をしておけばよかったかを考えてもらいます。

---

## ケース3：誰が動く？

これは個人的に考察して一番長倉さんらしいと思った次第です。

#### ケース

障害が発生しました。

顧客：「すぐ直してください」  
ベンダー「再現しません」 
社内：「原因分からない」  

あなたなら最初に何を整理しますか？

ここで出てくるのは

- 現象
- 再現条件
- 担当
- 判断者
- エスカレーション

一番効果が高いワークでかなと。

---

## 「誰の責任？」

一覧表を作ります。

| 出来事 | 顧客 | PM | ベンダー | 開発 |
|--------|:---:|:--:|:-------:|:---:|
| 要件決定 | ○ | ○ |  |  |
| 実装 |  |  | ○ | ○ |
| テスト | ○ | ○ | ○ | ○ |
| リリース判断 | ○ | ○ |  |  |

そして「この案件では誰が責任を持つと思いますか？」を考えてもらう。

最後に長倉さんが「実は責任は一人じゃなく、場面ごとに変わる」  
とまとめると、責任境界の理解につながります。

---

## まとめ

ここから先は個人的な意見ですので、参考にするしないはお任せします。

長倉さんらしい締め最後に一言だけ置くなら、

「プロジェクトで問題が起きる原因は、技術ではなく、責任と認識の境界が曖昧になることが多いです。  
だからこそ、誰が何を判断するのかを最初に整理することが重要です。」

この一文があると、ケーススタディ・ワーク・講義が一本の線でつながります。

私ならさらに一歩踏み込みます

これまで分析してきた内容を踏まえると、  
長倉さんのワークで育てたいのは「正解を知っている人」ではなく、  
「**境界が曖昧なときに、何を確認すべきかという問いを立てられる人**」

そのため、ワークの最後に必ず「**このケースで最初に確認すべきことは何ですか？**」  
という問いを置くことをおすすめします。

受講者は「正しい答え」を探すのではなく、  
「プロジェクトを成立させるために何を明らかにすべきか」という思考に切り替わるかなと思ってます。  

これは、長倉さんが伝えたい責任境界やプロジェクト成立条件とも自然につながる構成になると思います。
