作成日：2026年6月9日  
用途：**ISMS教育資料**（新任担当者向け）  
対象資料：A-02 ISMSマニュアル v2.1 / A-03 情報セキュリティ手順書 v2.1 / A-10 ISMS数値評価基準 / B-06-01 資産目録 / B-06-02 リスクアセスメント報告書 / ISMSトレーサビリティマトリクス

## 本資料の読み方

リスクNoとは「なぜ、ある資産にそのリスクが付与されたのか」を示すIDです。  
資産目録のリスクNoは「その資産を守るために必要な管理の種類」を意味し、  
リスクアセスメント表では「そのリスクがどの程度深刻か」を数値化しています。  

本資料では各リスクNoについて以下の7つの観点を解説します。

| 観点 | 内容 |
|-----|------|
| ① 本来の意味 | そのリスクNoが何を示す管理カテゴリか |
| ② 保護するCIA | 機密性(C)・完全性(I)・可用性(A)のどれを守るか |
| ③ 適用される資産 | どのような資産に付与されるか |
| ④ 発生の原因 | なぜそのリスクが生まれるのか |
| ⑤ 発生時の影響 | リスクが現実になった場合、何が起こるか |
| ⑥ 管理策が必要な理由 | なぜこの管理策が必要か |
| ⑦ 手順書が存在する理由 | A-03でその項番が定められている意味 |

## 主要リスクNo 詳細解説（重点12件）

> ★ 印は特に重点的に解説するリスク（重要資産に広く適用される中核リスク）

### ★ c1 ― 脅威インテリジェンスの不足（A5.7）

| 項目 | 内容 |
|-----|------|
| **管理テーマ** | 脅威情報の収集・分析体制の整備 |
| **保護するCIA** | **機密性（C）** |
| **リスク概要** | 外部の攻撃手法・マルウェア・フィッシング等の最新脅威を把握しておらず、予防措置が後手になる |
| **なぜ選定されるのか** | 機密性3の資産（顧客情報・プロジェクトファイル等）は攻撃者の標的となりやすい。脅威の動向を知らなければ適切な防御ができず、事後対応になって被害が深刻化する |
| **発生すると何が起こるか** | 新種マルウェアへの対応遅延、標的型攻撃に気づかない、侵害後に初めて問題を認識する事態になる。機密情報漏洩・顧客信用失墜・契約失注リスク |
| **対象となる資産例** | 顧客情報、案件情報、見積書、提案書、プロジェクトファイル、スキルシート、スマートフォン、エンジニア（要員） |
| **関連管理策** | 脅威インテリジェンスを構築する（対応状況：⏳予定） |
| **関連手順書** | A-03 5.7 脅威インテリジェンス |

**② 保護するCIA の論拠：**  
攻撃者は機密性の高い情報を狙う。脅威情報がなければ攻撃パターンを把握できず、機密情報が漏洩するリスクが高まるため **C（機密性）** の保護に直結する。

**④ 発生の原因：**  
情報セキュリティ委員会が外部の脅威情報ソース（IPA・JPCERT等）を定期的に確認・分析する仕組みがなく、対策の優先順位が脅威の変化に追いつかない。

**⑥ 管理策が必要な理由：**  
「脅威インテリジェンス」があれば、リスクアセスメントへの反映・対策の前倒し・インシデント発生確率の低減が可能になる。なければISMS全体がインプットを欠いた状態になる。

### ★ c5 ― アクセス制御の未整備（A5.15）

| 項目 | 内容 |
|-----|------|
| **管理テーマ** | アクセス制御方針の整備・適用 |
| **保護するCIA** | **機密性（C）** |
| **リスク概要** | 業務上必要のない人員が重要情報にアクセスできる状態になっている |
| **なぜ選定されるのか** | 電子データで保管・サーバで管理される機密性3の情報資産は、アクセス制御がなければ誰でも参照・コピーできる。「必要な人だけが必要な情報にアクセスできる」原則が守られないと情報漏洩が起きる |
| **発生すると何が起こるか** | 内部不正・誤操作による情報漏洩、顧客情報・個人情報の外部流出、法令違反（個人情報保護法）、顧客との信頼関係の喪失 |
| **対象となる資産例** | 顧客情報、スキルシート、案件情報、見積書、提案書、プロジェクトファイル、各仕様書・ソース |
| **関連管理策** | アクセス制御方針を整備（✅済） |
| **関連手順書** | A-03 5.15 アクセス制御 / 5.18 アクセス権 / 8.3 情報へのアクセス制限 |

**④ 発生の原因：**  
アクセス権の付与・剥奪ルールが文書化されておらず、退職・異動時の削除漏れや、部署外の人員への誤付与が発生しやすい環境にある。

**⑥ 管理策が必要な理由：**  
「最小権限の原則」は情報セキュリティの基本であり、アクセス権一覧を整備・定期棚卸しすることで内部不正と外部侵害後の横断移動を防ぐ。

### ★ c6 ― 利用者登録・削除ルールの不備（A5.16）

| 項目 | 内容 |
|-----|------|
| **管理テーマ** | IDライフサイクル管理（登録・変更・削除） |
| **保護するCIA** | **機密性（C）** |
| **リスク概要** | アカウントの作成・削除手順が明確でなく、不要なアカウントが残存する |
| **なぜ選定されるのか** | 退職者・異動者のアカウントが削除されないまま残ると、元従業員や第三者に悪用されるリスクがある。電子情報資産を保有する全組織に本質的なリスク |
| **発生すると何が起こるか** | 退職者による不正アクセス・情報持出し、共有アカウントによる責任の不明確化、認証突破後の内部情報への無制限アクセス |
| **対象となる資産例** | 顧客情報、スキルシート、プロジェクトファイル、各仕様書・ソース、案件情報・見積書（電子） |
| **関連管理策** | 利用者登録・削除ルールを整備（✅済） |
| **関連手順書** | A-03 5.16 識別情報の管理 / 5.18 アクセス権 |

**④ 発生の原因：**  
部門長が異動・退職のたびにシステム管理者へ申請する仕組みが形骸化していたり、申請書式が存在しない場合に発生する。

**⑥ 管理策が必要な理由：**  
「いつ・誰のアカウントが存在するか」を管理することはc5（アクセス制御）の実効性を担保する前提条件。アカウント管理なきアクセス制御は形骸化する。

### ★ c7 ― パスワードルールの未整備（A5.17）

| 項目 | 内容 |
|-----|------|
| **管理テーマ** | 認証情報（パスワード）の強度・管理ルール |
| **保護するCIA** | **機密性（C）** |
| **リスク概要** | 弱いパスワードや使い回しにより、不正ログインが成立しやすい状態にある |
| **なぜ選定されるのか** | 機密性3の電子情報資産はシステム認証で守られている。認証を突破されれば全てのアクセス制御が無効化される。パスワードは最初の防衛線 |
| **発生すると何が起こるか** | 不正ログインによる情報漏洩・改ざん、フィッシング被害、ブルートフォース攻撃の成功、クレデンシャルスタッフィング攻撃の被害 |
| **対象となる資産例** | 顧客情報、スキルシート、プロジェクトファイル、各仕様書・ソース、案件情報・見積書（電子） |
| **関連管理策** | 利用者パスワードルールを整備（✅済） |
| **関連手順書** | A-03 5.17 認証情報 / 8.5 セキュリティを保った認証 |

**④ 発生の原因：**  
パスワードの最低文字数・複雑性・定期変更の義務が文書化されておらず、担当者ごとに設定が異なる。特に管理者IDのパスワードが初期値のままになるリスクがある。

**⑥ 管理策が必要な理由：**  
A-03 5.17では「8文字以上・英数字大小混在」を規定している。これはc5（アクセス制御）・c6（利用者管理）と三位一体で機密情報を守る認証体系を形成する。

### ★ c17 ― インシデント報告ルールの未整備（A6.8）

| 項目 | 内容 |
|-----|------|
| **管理テーマ** | インシデント・セキュリティ弱点の報告体制 |
| **保護するCIA** | **機密性（C）** |
| **リスク概要** | 情報セキュリティ事象が発生しても報告経路が不明確で、対応が遅延する |
| **なぜ選定されるのか** | インシデント発生時に報告が遅れると被害が拡大する。特に機密性3の資産では初動対応の遅れが漏洩範囲を指数的に拡大させる。報告ルールは「予防」ではなく「被害最小化」のための管理 |
| **発生すると何が起こるか** | 発見者が「自分で解決しよう」と試みて対応を誤る、委員長への報告が数日遅れる、外部（顧客・監督機関）への通知がさらに遅延し法的問題に発展する |
| **対象となる資産例** | 顧客情報、案件情報、見積書、提案書、プロジェクトファイル、スキルシート、スマートフォン、エンジニア（要員） |
| **関連管理策** | インシデント・弱点の報告ルールを整備（✅済） |
| **関連手順書** | A-03 5.24 インシデント管理の計画策定 / 5.25 事象の評価及び決定 / 5.26 インシデントへの対応 / 6.8 情報セキュリティ事象の報告 |

**④ 発生の原因：**  
「誰に報告するのか」「何がインシデントなのか」が従業者に周知されていない場合、事象の発見者が自己判断で処理しようとして初動が遅れる。

**⑥ 管理策が必要な理由：**  
A-10のインシデントレベル定義（Level 0〜2）と組み合わせることで、被害範囲に応じたエスカレーションが機能する。報告ルールはISMSの「神経系」ともいえる。

### ★ c31 ― 不要情報の削除不備（A8.10）

| 項目 | 内容 |
|-----|------|
| **管理テーマ** | 不要になった情報・媒体の確実な削除・廃棄 |
| **保護するCIA** | **機密性（C）** |
| **リスク概要** | 業務上不要になった機密情報が削除されずにサーバ・端末・媒体に残存し、漏洩経路になる |
| **なぜ選定されるのか** | 電子データは「使わなくなっても消えない」。機密性3の資産が不要になった後も残存すれば、攻撃者やアクセス権のある関係者による不正利用のリスクが恒常的に存在する |
| **発生すると何が起こるか** | 廃棄予定の端末から顧客情報・個人情報が復元される、退職者が過去データへアクセスできる状態が続く、データ侵害時の被害範囲が「現在使用中＋過去データ」に拡大する |
| **対象となる資産例** | 顧客情報、スキルシート、案件情報、見積書・提案書（電子）、プロジェクトファイル、各仕様書・ソース |
| **関連管理策** | 情報が不要になった場合の削除する仕組みを導入（⏳予定）　※リスクレベル12・最高水準 |
| **関連手順書** | A-03 8.10 情報の削除 / 7.14 装置のセキュリティを保った処分又は再利用 |

**④ 発生の原因：**  
「削除するタイミング」「削除の承認フロー」「削除の記録」が規定されておらず、担当者任せになっている。プロジェクト終了後や退職者発生後のデータ整理が行われない。

**⑥ 管理策が必要な理由：**  
「不要な情報を持たない」ことが最強の漏洩対策。削除仕組み・廃棄記録を整備することで、保有データの最小化（data minimization）を実現し監査時の説明責任も果たせる。

### ★ c33 ― データ漏洩防止の未整備（A8.12）

| 項目 | 内容 |
|-----|------|
| **管理テーマ** | DLP（Data Loss Prevention）対策・通信の監視 |
| **保護するCIA** | **機密性（C）** |
| **リスク概要** | 機密情報がネットワーク・メール・USBメモリ等を経由して外部へ流出することを検知・防止する仕組みがない |
| **なぜ選定されるのか** | アクセス制御・パスワード管理が整備されていても、「内部からの持出し」は防げない。機密性3の電子情報資産には、外部流出を技術的に防ぐ層が必要 |
| **発生すると何が起こるか** | 内部者による意図的な持出し、誤メール送信による個人情報流出、USBメモリ経由での情報持出し、顧客・規制当局への通知義務発生、損害賠償リスク |
| **対象となる資産例** | 顧客情報、スキルシート、プロジェクトファイル、各仕様書・ソース、案件情報・見積書（電子） |
| **関連管理策** | データ漏洩防止対策を明確化・実施（⏳予定）　※リスクレベル12・最高水準 |
| **関連手順書** | A-03 8.12 データ漏洩防止 / 5.14 情報の転送 |

**④ 発生の原因：**  
「外部へ送信してよい情報かどうか」の技術的チェックがなく、従業者の判断だけに依存している。ネットワーク監視・端末の暗号化が未実装の場合、漏洩は気づかれない。

**⑥ 管理策が必要な理由：**  
DLP対策はc5（アクセス制御）が「入口の鍵」とすれば「出口の鍵」にあたる。多層防御の最終ラインとして、内部・外部双方からの流出を防ぐ。

### ★ c35 ― 監視活動の不足（A8.16）

| 項目 | 内容 |
|-----|------|
| **管理テーマ** | ネットワーク・システム・アプリケーションの継続的監視 |
| **保護するCIA** | **機密性（C）** |
| **リスク概要** | 不正アクセス・異常通信・マルウェア活動を検知する仕組みがなく、インシデントが長期間気づかれない |
| **なぜ選定されるのか** | 全ての予防策をすり抜ける攻撃は存在する。機密性3の資産を守るためには「侵害を早期発見する能力」が不可欠。検知が遅れると被害は指数的に拡大する |
| **発生すると何が起こるか** | 侵害が数週間〜数ヶ月気づかれない、大量の顧客情報が流出した後に発覚、事後の原因究明が困難（ログなし）、証拠保全不能による法的対応の遅延 |
| **対象となる資産例** | 顧客情報、スキルシート、プロジェクトファイル、各仕様書・ソース、案件情報・見積書（電子） |
| **関連管理策** | ネットワーク・システム・アプリケーションの異常動作を監視（⏳予定）　※リスクレベル12（起こりやすさ4×影響3）・最高水準 |
| **関連手順書** | A-03 8.16 監視活動 / 8.15 ログ取得 |

**④ 発生の原因：**  
ログ取得・監視ツールが未整備、または取得していても定期的なレビューがなく「ログは取っているが見ていない」状態になっている。

**⑥ 管理策が必要な理由：**  
「防御」と「検知」は車の両輪。c5・c7が防御側とすれば、c35は検知側。ISMS審査でも監視体制の有無は重要な確認ポイント。

### ★ c41 ― ウェブフィルタリングの不備（A8.23）

| 項目 | 内容 |
|-----|------|
| **管理テーマ** | 業務端末からの不適切なウェブサイトへのアクセス制限 |
| **保護するCIA** | **機密性（C）** |
| **リスク概要** | マルウェア配布サイト・フィッシングサイトへのアクセスが制限されておらず、感染・情報漏洩の入口になる |
| **なぜ選定されるのか** | 電子データを扱う端末が悪意あるウェブサイトに接続することで、マルウェア感染・クレデンシャル盗取が起こり得る。機密性3の電子資産を保有する全端末に適用される |
| **発生すると何が起こるか** | ドライブバイダウンロードによるマルウェア感染、フィッシングによるパスワード盗取、C2サーバ通信による情報外部送信、ランサムウェア感染による業務停止 |
| **対象となる資産例** | 顧客情報、スキルシート、プロジェクトファイル、各仕様書・ソース、案件情報・見積書（電子） |
| **関連管理策** | 外部ウェブサイトへのアクセスを管理（⏳予定） |
| **関連手順書** | A-03 8.23 ウェブフィルタリング / 8.7 マルウェアに対する保護 / 8.20 ネットワークのセキュリティ |

**④ 発生の原因：**  
ウェブフィルタリングシステムが未導入、または導入済みでも禁止リストの更新・管理が行われていない。

**⑥ 管理策が必要な理由：**  
ウェブフィルタリングは「入口対策」の一環。c35（監視）が異常を検知するとすれば、c41は危険な接続そのものを遮断する予防策。

### ★ i5 ― セキュリティ構成管理の不備（A8.9）

| 項目 | 内容 |
|-----|------|
| **管理テーマ** | ハードウェア・ソフトウェア・サービスのセキュリティ設定管理 |
| **保護するCIA** | **完全性（I）** |
| **リスク概要** | サーバ・PC・ネットワーク機器のセキュリティ設定が管理されておらず、脆弱な設定が放置される |
| **なぜ選定されるのか** | 設定ミスは「意図しないセキュリティホール」を生む。完全性3の電子資産（各仕様書・ソースコード等）はシステムの設定が正確であることに依存しており、設定の誤りは改ざん・破損のリスクになる |
| **発生すると何が起こるか** | 設定ミスによる権限昇格・不正アクセス、パッチ未適用による既知脆弱性の悪用、設定変更が追跡できず原因究明が困難、開発環境と本番環境の設定乖離によるセキュリティ事故 |
| **対象となる資産例** | 各仕様書・ソース・プロジェクトファイル・テストデータ（電子）、顧客情報、スキルシート、保守契約書 |
| **関連管理策** | セキュリティ構成（Configuration）を管理（⏳予定） |
| **関連手順書** | A-03 8.9 構成管理 / 8.8 技術的ぜい弱性の管理 / 8.19 運用システムへのソフトウェアの導入 |

**④ 発生の原因：**  
構成管理台帳が存在せず、誰がどの設定を変更したかが追跡できない。セキュリティパッチの適用状況も把握されていない。

**⑥ 管理策が必要な理由：**  
構成管理は変更管理（i9）と連動し、「あるべき状態」を定義することで逸脱の検知を可能にする。特に開発環境を持つ組織では必須。

### ★ i9 ― 変更管理ルールの未整備（A8.32）

| 項目 | 内容 |
|-----|------|
| **管理テーマ** | システム・文書・設定の変更プロセス管理 |
| **保護するCIA** | **完全性（I）** |
| **リスク概要** | 変更管理のルールが明確でなく、意図しない変更や古い情報での業務実施が起こる |
| **なぜ選定されるのか** | 完全性3の資産（仕様書・ソース・契約書・ISMS記録等）は「正確な最新版」が存在することが価値の前提。変更ルールがなければ、古い版での業務や無承認変更が発生する |
| **発生すると何が起こるか** | 古い仕様書に基づく開発ミス、契約書の版管理失敗による法的紛争、ISMS記録の改ざん・誤記録、システム変更の影響把握失敗による業務停止 |
| **対象となる資産例** | 各仕様書・ソース・テストデータ・プロジェクトファイル・保守契約書・ISMS関連記録・顧客情報 |
| **関連管理策** | 変更管理のルールを明確化（✅済） |
| **関連手順書** | A-03 8.32 変更管理 / 8.9 構成管理 |

**④ 発生の原因：**  
変更の申請・承認フロー・変更履歴の記録が文書化されておらず、担当者の判断で変更が行われている。版管理がされておらず、最新版の特定が困難な状態。

**⑥ 管理策が必要な理由：**  
「変更の追跡可能性」は完全性の核心。変更管理記録はi5（構成管理）と連動して「変更前後の差分」を明確にし、問題発生時のロールバックを可能にする。

### ★ a6 ― 事業継続のためのICTの備え不足（5.30）

| 項目 | 内容 |
|-----|------|
| **管理テーマ** | ICTシステムの事業継続計画（BCP）整備 |
| **保護するCIA** | **可用性（A）** |
| **リスク概要** | 自然災害・システム障害発生時のICT復旧計画・目標が未整備で、長期業務停止に陥る |
| **なぜ選定されるのか** | 可用性3のプロジェクトファイルは、利用不能になると業務・顧客納期に直接影響する。ICTの備えは「予防」ではなく「被害からの回復速度」を決める |
| **発生すると何が起こるか** | 災害・障害後にITシステムが1週間以上復旧しない、顧客への納品が遅延・中断、事業継続不能による契約違約金・顧客離反、競合他社への案件流出 |
| **対象となる資産例** | プロジェクトファイル（A=3）、ファイルサーバ・ネットワーク機器・UTM（A=3）、開発環境（A=3） |
| **関連管理策** | 事業継続のためのICTの備えを計画・テスト（⏳予定）　※リスクレベル8（起こりやすさ2×影響4） |
| **関連手順書** | A-03 5.30 事業継続のためのICTの備え / 5.29 事業の中断・阻害時の情報セキュリティ |

**④ 発生の原因：**  
RTO（目標復旧時間）・RPO（目標復旧時点）が設定されておらず、バックアップはあっても「どのくらいで復旧できるか」のシナリオが未検証。

**⑥ 管理策が必要な理由：**  
ISMS審査では「可用性を守るための継続計画」の存在が求められる。ICT継続計画がなければ、火災・水害・ランサムウェア被害の際に復旧の見通しが立たない。

## まとめ：新任担当者のための3つの理解ポイント

### ポイント1：リスクNoは「守りたいもの」によって決まる

| CIAが高い場合 | 付与されるリスクNo群 |
|-------------|------------------|
| 機密性（C）= 3 | c1, c3〜c8, c14, c15, c17, c18, c21, c22, c31〜c33, c35〜c37, c41 |
| 完全性（I）= 3 | i1, i5, i6, i9 |
| 可用性（A）= 3 | a1, a3, a6, a9 |

### ポイント2：リスクNoはカテゴリ別に覚えると理解しやすい

| カテゴリ | 関連リスクNo | 一言で言うと |
|--------|------------|------------|
| 認証・アクセス | c5, c6, c7 | 「誰が入れるか」を管理する |
| 情報の扱い | c2, c3, c4, c22, c31 | 「情報をどう扱うか」を管理する |
| 外部脅威対策 | c1, c18, c21, c33, c35, c41 | 「外からの攻撃を防ぐ・検知する」 |
| 人的管理 | c14, c15, c17 | 「人のミス・悪意を防ぐ」 |
| 供給者管理 | c8, c36, c37 | 「外部委託・ソフトウェアを制御する」 |
| データ整合性 | i1, i5, i6, i9 | 「情報が正確であり続ける」 |
| 事業継続 | a1, a3, a6, a9 | 「使い続けられる状態を維持する」 |

### ポイント3：対応が未完了（⏳）のリスクは特に注意

対応予定のリスクは管理策が整備されておらず、証跡もない状態を意味します。  
監査・更新審査では「なぜ未完了か」「いつ完了するか」の説明が求められます。

| ⏳予定リスク（計11件） | 優先度 |
|--------------------|--------|
| c35, c33, c31, c1 | 🔴 最高（リスクレベル12） |
| c18, c32, c41, i5, c2 | 🟡 高（リスクレベル9） |
| a6 | 🟡 高（リスクレベル8・影響=4） |
| a9 | 🟢 中（リスクレベル6） |
