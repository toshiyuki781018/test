# クラウドネイティブ勉強会　Lecturer 向けガイドライン v1.0

*クラウドネイティブ教科書の設計思想を、講義へ継承するための運用文書*

---

## 本ガイドラインについて（はじめに）

このガイドラインは、**講義資料ではありません**。  
Lecturer（講師）が講義資料を作成する**前に**読む、運用文書です。

クラウドネイティブ教科書は、操作や手順を覚えるための本ではありません。  
「変化する環境の中で、システムをどう成立させ続けるか」を  
**構造として理解する**ための本です。  

この思想を、講師が変わっても、担当章が変わっても、  
勉強会全体で一貫して受講者へ届けることが、このガイドラインの狙いです。

一方で、教科書は「唯一の正解」を教える本でもありません。  

だからこそ、**伝えるべき本質は共通化しつつ、説明の方法は  
Lecturer ごとの個性を活かせる形**を目指します。  

この2つの両立を、本書では「Must（必須）」と「How（任意）」  
という考え方で整理します。

作成にあたっては、以下のプロジェクト資料を  
横断的に参照しています。

- クラウドネイティブ教科書 本文（序章〜第6部、付録）
- Illustrated（図版）
- 勉強会企画資料（開催概要・進行構成案）
- 打ち合わせ議事録
- KCNA 関連資料
- プロジェクト内の設計方針資料

> なお、英語翻訳プロジェクトは参照対象から除外しています。  
> 本ガイドラインは、**日本語版教科書を正式版**として作成しています。

### 用語について

本ガイドラインでは、プロジェクトで用いられている正式名称として、  
以下を使用します。

| 呼称 | 本ガイドラインでの表記 |
|---|---|
| 講師 | **Lecturer** |
| 図版 | **Illustrated** |

## 1. 本ガイドラインの目的

### 1.1 なぜガイドラインが必要か

勉強会は、単発イベントではなく、クラウドネイティブ技術を学ぶ  
**コミュニティ形成の場**として位置付けられています。  

複数の Lecturer が章を分担するため、  
放っておくと説明の粒度・強調点・言葉づかいが人によってばらつきます。

そのばらつきには、良いものと悪いものがあります。

- **残したいばらつき**：例え話、デモの見せ方、現場事例、深掘りの角度など、  
Lecturer の経験や個性から生まれる「How」の部分。
- **抑えたいばらつき**：「その技術がなぜ生まれたのか」「何を解決するのか」という  
**設計思想＝Must の部分**が、人によって伝わったり伝わらなかったりすること。

このガイドラインは、後者を防ぎ、前者を活かすためにあります。

### 1.2 このガイドラインが目指すこと

このガイドラインは、次の5つを達成するために作られています。

1. **Kubernetes 操作本にしない**   
— 手順の暗記ではなく、構造の理解を届ける。
2. **クラウドネイティブを構造として理解してもらう**  
— 個別技術の寄せ集めとしてではなく、一つの流れとして捉えられるようにする。
3. **技術の背景を理解してもらう**  
— 「なぜその技術が生まれたのか」から説明する。
4. **KCNA 学習へ接続する**  
— 資格対策講座にはしないが、構造理解が   
KCNA の知識体系へ自然につながるようにする。
5. **初学者の理解を支援する**  
— 分からないことを前提に、往復しながら積み上げられるようにする。

### 1.3 このガイドラインの使い方

Lecturer は、担当章の資料を作り始める前に、  
まず「2. 教育方針」と「4. Must / How の考え方」を読んでください。  

そのうえで、担当回の「5. 各回ごとの Must」と、扱う   
Illustrated について「6. Illustrated 解説の観点」を確認します。  

資料が一通りできたら、「8. Lecturer セルフチェック」で自己点検してください。

## 2. クラウドネイティブ教科書の教育方針

Lecturer が最初に共有すべき、教科書全体を貫く方針です。  
個々の章を教える前に、この土台を理解してください。

### 2.1 「前提の変化」として捉える

クラウドネイティブは、新しい技術だから学ぶのではありません。  
**従来の前提では、システムが成立しなくなったために必要になった**ものです。

- モノリスは、固定された前提のもとで成立していた。
- 仮想化は、環境を分離したが、前提は変えなかった。
- コンテナは、単位を小さくしたが、全体は人が管理していた。
- クラウドネイティブでは、**前提そのものが変わった**。

変化・分散・不確実性が「例外」ではなく  
「前提」になった——この一点が、教科書全体の出発点です。  

Lecturer はこれを、技術の進化の話としてではなく、  
**前提条件の変化**の話として語ってください。

### 2.2 「静的最適化」から「動的安定」へ

従来の設計は「変化しないこと」を前提に最適化されていました。  
クラウドネイティブでは前提が逆転します。  

**壊れないことも、完全に制御できることも前提にしません**。  
一部が壊れ、状態がずれ、環境が変わる  
——それでも全体として成立し続けることを目指します。

「止めないこと」を目指すのではなく、「止まっても全体が成立する」ようにする。  
**運用（人の頑張り）で支えるのではなく、仕組みで吸収する。**  

この考え方が、宣言的制御・可観測性・GitOps などすべての技術の根にあります。

### 2.3 構造で理解するとは、「責任の置き場所」を見ること

教科書は「なぜその形になっているのか」を、ツールや手順より先に扱います。  

構造で理解するとは、技術そのものではなく、  
**その技術が引き受けている責任を見る**ことです。

教科書全体で変わらない3つの問いがあります。  

Lecturer はこの問いを、  
章をまたいで繰り返し受講者に投げかけてください。

> - 何を成立させるのか。
> - どこまで責任を持つのか。
> - どこで境界を引くのか。

### 2.4 教科書の構造には順番の意味がある

序章で視点を渡し、  
第1部で原理を理解し、  
第2部で体感し、  
第3部で構造として扱い、  
第4部で運用として判断し、  
第5部で社会と接続し、  
第6部で総括する

この順番には意味があります。  
**前の部で整理した前提の上に、次の部が成り立ちます。**

Lecturer は自分の担当回を独立した講義として完結させず、  
「前回までに積んだ前提の上に、今回がある」ことを意識して、  
冒頭で前提を引き継ぎ、末尾で次への問いを渡してください。

### 2.5 「理解した気になる説明」を避ける

教科書は、あえて分かりやすく単純化しすぎることを避けています。  

すぐに分からなくても構わない、先に進んでから戻ると見え方が変わるという  
**往復による理解**を前提にしています。  

Lecturer も、その場で全部を分からせようと詰め込むより、  
「今は分からなくてよい。後でつながる」と伝える余裕を持ってください。

## 3. Lecturer の役割

### 3.1 ミッション

**クラウドネイティブエンジニアの育成支援。**

Lecturer に求められるのは、技術知識だけではありません。  

**技術の生い立ちや背景から説明できること**が重視されます。  
具体的には、次のような問いに答えられることが重要です。

- なぜ VM からコンテナへ移行したのか。
- Kubernetes はなぜ必要になったのか。
- マイクロサービス化で何が変わったのか。

### 3.2 具体的な役割

Lecturer の役割は2つあります。

1. **クラウドネイティブ勉強会の Lecturer**   
— 担当章について、教科書の思想を継承した講義を行う。
2. **クラウドネイティブ教科書の開発プロジェクトへの参加**   
— 教科書のアップデートにも積極的に関わる。

### 3.3 対象者と姿勢

主な対象は、  
**これから学習を始める初級エンジニア（Linux の基本知識あり）**と、  
**KCNA 受験を検討・学習中のエンジニア**です。

初学者に対して、分かりやすく丁寧に接すること。  
教えることへの熱意を持つこと。  

これが Lecturer に共通して求められる姿勢です。  

受講者の「分からない」を能力の問題ではなく、  
まだ構造として捉える視点を持てていないだけと捉えてください。

### 3.4 学習環境について

ハンズオンは **Killercoda** を使用します。  

Google アカウントで利用でき、事前構築不要で Kubernetes 環境をすぐに使えるため、  
参加者の環境差異によるトラブルを最小化できます。  

参加者には事前にアカウント連携のみ実施してもらう想定です。  

Lecturer は環境構築の説明に時間を取られず、  
**構造の理解に時間を使える**ことを前提に設計してください。

## 4. Must / How の考え方

このガイドラインの中核です。**Must と How を区別すること**が、  
「一貫性」と「個性」を両立させる仕組みです。

### 4.1 Must（必須事項）— 共通化する本質

Must は、どの Lecturer が教えても**必ず伝えなければならない本質**です。  
ここがぶれると、勉強会全体のメッセージが崩れます。  

Must には次のものが含まれます。

- 各章で伝えるべき**背景**（なぜその技術が生まれたのか）
- **学習目標**（受講者が何を持ち帰るのか）
- **必須メッセージ**（この回で絶対に残す一言）
- **誤解しやすいポイント**（放置すると誤った理解につながる点）
- **Illustrated（図版）説明の観点**（図で何を伝えるのか）

### 4.2 How（任意事項）— 個性を活かす方法

How は、本質をどう伝えるかという**方法**であり、  
Lecturer の自由と個性に委ねられます。  

How には次のようなものがあります。

- **デモ内容**（何を、どう動かして見せるか）
- **実務経験の共有**（自身の現場で起きたこと）
- **補足資料**（参考リンク、追加の図）
- **現場事例**（具体的なトラブルや設計判断の例）
- **深掘り内容**（興味を持った受講者向けの発展）

### 4.3 判断基準：Must か How かの見分け方

迷ったときは、次の問いで判断してください。

> **「これを伝えないと、受講者が構造を誤解する / 技術を操作手順として覚えてしまうか？」**  
> → **Yes なら Must。No なら How。**

例えば「Kubernetes は現在の状態とあるべき状態の差分を埋め続ける」は Must です。  
これを飛ばすと、Kubernetes を単なるコンテナ起動ツールと誤解します。  

一方、「その差分制御を、自分の現場ではこう活用した」という体験談は How です。  
あってもなくても本質は崩れません。

### 4.4 やってはいけないこと

- Must を省略して How（デモや事例）だけで時間を使い切ること。
- Must を、自分の解釈で本質と異なる内容に書き換えること。
- How を Must のように「これが唯一の正解」として教えること  
（クラウドネイティブに絶対の正解はありません）。

## 5. 各回ごとの Must

勉強会は**全4〜5回**を想定しており、  
回数と割り当ては引き続き検討中（要検討）です。  

現時点の候補構成に沿って、  
各回で**必ず伝える背景・学習目標・必須メッセージ・誤解ポイント**を示します。  

担当回の資料は、この Must を必ず満たしたうえで、  
How を上乗せして構成してください。

各回に共通する冒頭・末尾の型は次の通りです。  

**冒頭で「前回までに積んだ前提」を1〜2分で引き継ぎ、  
末尾で「次に残る問い」を渡す。**  

教科書がすべての部末で「次に残る問い」を置いているのと同じ構造です。

### ■ 第1回 ｜ 序章＋第1部：基礎ステップ（前提と実行単位）

　**▼ 背景（Must）**  
　モノリス → VM → コンテナ → クラウドネイティブという流れを、  
　性能向上の歴史ではなく、**前提が変わってきた歴史**として語る。  

　コンテナで単位が小さくなり分散した結果、  
　「増え続ける単位を、誰がどう全体として成立させ続けるのか」という問いが生まれ、  
　そこに Kubernetes が要請された、という順で背景を通す。

　**▼ 学習目標（Must）**  
　クラウドネイティブを個別技術の寄せ集めではなく、  
　**単位・制御・実行環境・前提条件という構造**として捉えられるようになる。

　「どこで何が起きているのかを切り分けられる」視点を持つ。

　**▼ 必須メッセージ（Must）**  
- 変化・分散・不確実性は「例外」ではなく「前提」である。  
- Kubernetes は「どうするか」ではなく「どうなっていてほしいか」を定義し、  
**差分を埋め続ける**仕組みである（宣言的制御）。
- 資源は有限、ネットワークは不確実、失敗は避けられない  
——これらは消えたのではなく、見えなくなっているだけ。

**▼ 誤解しやすいポイント（Must）**
- 「Kubernetes ＝ コンテナを起動するツール」  
→ 実際は**あるべき状態を保ち続ける制御の仕組み**。
- 「Kubernetes は OS のようなもの」  
→ 資源を直接管理するのではなく、その上で状態を成立させ続ける**層**。OS ではない。
- 「クラウドが常に正解」  
→ クラウドとオンプレは特徴が違うだけで、**クラウドは絶対的な正解ではない**（第1部3章）。

### ■ 第2回 ｜ 第2部：実践ステップ（前提を体感する / Killercoda ハンズオン）

　**▼ 背景（Must）**  
　第1部で「構造として分かった」ことは、言葉だけでは定着しない。  
　**手を動かすと思った通りに動かない**  

　——その体感を通じて、前提が実際の挙動としてどう現れるかを確認するのがこの回の目的。

　**▼ 学習目標（Must）**

- Pod
- Deployment
- Service
- ConfigMap
- Secret
- Volume
- Ingress
- kubectl

　これらの基本操作を通じて、「**操作を覚える**」のではなく  
　「**何が起きているのかを見る視点**」を体で捉える。

　**▼ 必須メッセージ（Must）**
- この回の目的は操作の暗記ではない。  
　**第1部で整理した前提が、実際の挙動としてどう現れるか**を確認すること。
- Pod を消したら別の場所で起動した、設定を変えたのに反映されない  
　——これは操作ミスではなく、**宣言的制御が「ずれ続ける前提で戻し続けている」姿**である。
- 壊れることは例外ではなく、**戻ることが前提**。  
　人が一つずつ操作するのではなく、あるべき状態を定義して成立を仕組みに任せる。

　**▼ 誤解しやすいポイント（Must）**
- 「kubectl コマンドを覚えることがゴール」  
→ ハンズオンは**挙動の観察**が目的。コマンドは手段。
- 「エラー＝自分の操作ミス」  
→ どこで何が起きているかを**切り分ける**練習の場と捉える。
- ハンズオンに寄せすぎて「操作本」化しないよう、常に第1部の構造と結びつける。

### ■ 第3回 ｜ 第3部：応用ステップ（分割・変更・通信）※要検討

　**▼ 背景（Must）**  
　アプリケーションを分割すると、  
　システムの中に「**変更という流れ**」と「**通信という流れ**」が生まれる。  

　この2つの流れをどこで引き受けるのか、という問いが第3部のテーマ。

　**▼ 学習目標（Must）**  
　マイクロサービス・GitOps・サービス間通信制御を、  
　「選ぶべき正解」としてではなく**判断の積み重ね**として捉える。  

　「何を選ぶか」より「**何を引き受けるかを自覚しているか**」が重要だと理解する。

　**▼ 必須メッセージ（Must）**  
- **複雑さは消えない。置き場所が変わるだけ。**   
分割は複雑さを減らす行為ではなく、通信・依存・状態を外に出す行為。
- **マイクロサービスは万能ではない。**   
目指すべき理想形ではなく、**設計上の選択肢**。前提が変われば最適な形も変わる。
- GitOps は判断をなくすものではなく、**判断を構造として固定する**考え方。  
人が毎回判断しなくても成立する状態を作る（ツール＝ArgoCD/Flux は手段にすぎない）。

　**▼ 誤解しやすいポイント（Must）**  
- 「マイクロサービス化＝正しい/進んでいる」  
→ 分割にはコスト（通信の遅延・失敗、状態のズレ、依存の拡大）が伴う。
- 「GitOps ＝ 自動デプロイ」「継続的デリバリー ＝ 自動化」  
→ 本質は**再現性と、いつでも同じ手順で戻せること**。
- 「ツール（ArgoCD, Istio 等）を入れれば解決」  
→ 重要なのは**構造が成立していること**。ツールの違いではない。

### ■ 第4回 ｜ 第4部：運用ステップ（可観測性と運用） ※要検討

　**▼ 背景（Must）**  
　分割によって判断を外に出しても、**分散したシステムの内部状態は直接は見えない**。  
　見えるのは結果としての挙動（遅い・エラー・停止）だけ。  

　原因を判断できなければ運用は成立しない——ここで可観測性が要請される。
 
　**▼ 学習目標（Must）**  
　「見えること」と「判断できること」は別であると理解し、  
　**観測できる情報から状態を推測する**という視点を持つ。

　運用とは「判断の連続」であると捉える。

　**▼ 必須メッセージ（Must）**  
- 可観測性とは**すべてを見ること**ではなく、**判断に必要な情報を扱える状態にすること**。  
そして「どこまで観測するか」を決める設計でもある（観測にもコストがある）。
- モニタリング（決めた指標の監視）と可観測性（観測情報から内部状態を推測）は役割が違う。
- 運用の本質は「見る → 追う → 制御する → 判断する」の連続。  
すべてを完璧にはできないからこそ、**どこまでやるかを決める力**が重要。

　**▼ 誤解しやすいポイント（Must）**  
- 「ログ・メトリクスを増やせば分かる」  
→ 情報が多くても、関連づけられなければ判断できない。増やしすぎは逆効果。
- 「Prometheus / Grafana / Jaeger を入れれば可観測性が手に入る」  
→ ツールは手段。**何を観測するかという判断**が本体。
- 「可観測性 ＝ 監視の高機能版」  
→ 目的は「見る」ではなく「**判断できる**」こと。

### ■ 第5回 ｜ 第5部：展開ステップ ＋ 第6部：総括（構造を社会・価値へ接続） ※要検討

　**▼ 背景（Must）**  
　第1〜4部で見てきた「責任の置き場所を決める構造」が、  
　働き方・価値の出し方・コスト（FinOps）・AI といった  
　社会やビジネスにどうつながるかを扱う。

　技術の話の先にある「**構造を扱う力**」で締めくくる。

　**▼ 学習目標（Must）**  
　同じ技術を扱っていても、**どこまでを自分の責任として引き受けるか**によって  
　提供する価値がまったく変わる、と理解する。

　技術が変わっても応用できる土台（構造を扱う力）を持ち帰る。

　**▼ 必須メッセージ（Must）**  
- コストは後から発生するのではなく、**構造がそのまま反映される**（FinOps）。  
AI もまた責任の分け方を変える技術で、構造の延長にある。
- 技術・手法・役割は変わり続けるが、変わらないものがある  
——**何を成立させるのか / どこまで責任を持つのか / どこで境界を引くのか**。
- 構造を扱えるようになると、技術に振り回されず、新しいツールが出ても  
「それが何を解決し、何を引き受けるのか」を判断できる。

　**▼ 誤解しやすいポイント（Must）**  
- 「クラウドネイティブを学べば正解の運用が手に入る」   
→ **クラウドネイティブは絶対的な正解ではない。** 変化の中で構造を捉え続けることがゴール。
- 「まとめ回＝これまでの要約」  
→ 総括は要約ではなく、「**自分は何を成立させ、どこまで責任を引き受けるのか**」という問いを受講者に返す回。

### ■ 5.x KCNA への接続（全回共通の Must）

　勉強会は**資格対策講座を主目的とはしません**。  
　しかし、本書の構造理解は KCNA の知識体系と接続しやすいように作られています。  

　Lecturer は各回で、扱った構造が KCNA のどの領域に対応するかを**一言添える**ことができます。

| 教科書の構成（勉強会） | KCNA の対応領域 |
|---|---|
| 序章・第1部 | Cloud Native Fundamentals / Kubernetes Fundamentals |
| 第2部 | Kubernetes Fundamentals / Orchestration |
| 第3部 | Cloud Native Architecture / Application Delivery |
| 第4部 | Observability |
| 第5部 | Cloud Native Concepts（関連領域） |

　伝えるべき Must は「**KCNA 用語を覚えること**」ではなく、  
　「**理解してきた構造に、知識体系としての名前を接続すること**」です。

## 6. Illustrated 解説の観点

Illustrated（図版）は、装飾ではなく  
**思想を一枚で伝えるための道具**です。  

Lecturer は図を「見せて終わり」にせず、  
**その図が何の前提を可視化しているか**を言葉にしてください。  

主要な図と、解説時に押さえる観点は次の通りです。

**■ 図 P-1 ｜ 技術進化ではなく「前提の変化」（序章）**  
　モノリス → VM → コンテナ → クラウドネイティブを、  
　性能の階段としてではなく**前提条件の断絶**として読ませる図。  

　「これは進化の話ではなく、前提が変わった話」と一言添える。

**■ 図 1-2.0 ｜ Kubernetes の宣言的制御ループ（第1部）**  
　「あるべき状態」と「現在の状態」の差分を見続け、埋め続けるループを示す図。  

　**一度合わせて終わりではなく、ずれ続ける前提で戻し続ける**ことを、  
　矢印が一周する動きとして解説する。宣言的制御の核。

**■ 図 1-2.1 ｜ Control Plane と Worker Node の役割分担（第1部）**  
　状態を扱う側と、実際に動かす側を分けた設計判断。  
　**一部が壊れても全体が崩れない**ための責務分割であることを強調する。

**■ 図 3-1 ｜ 複雑さは消えない — 置き場所が変わるだけ（第3部）**  
　分割前は内側にあった複雑さ（通信・変更・依存）が、分割後に外側に現れる様子を示す図。  
　**「分割＝複雑さの削減」という誤解を正すための一枚**。  
　この図はマイクロサービスの Must メッセージそのもの。

**■ 図 3-2 ｜ クラウドとオンプレミスの前提の違い / GitOps（第3部）**  
　変更が Git に記録され、差分として比較でき、戻せる  
　——**変更という不確実な流れを追える状態**にする構造を読ませる。

**■ 図 4-1.0 ｜ 分散した瞬間に全体が見えなくなる（第4部）**  
　情報は存在するのに、分断されていて「見えているのに分からない」状態を示す図。  
　可観測性が必要になる理由を、この一枚で腹落ちさせる。

**■ 図 4-1.1 ｜ モニタリングと可観測性の守備範囲（第4部）**  
　決めた指標を監視するモニタリングと、  
　観測情報から内部状態を推測する可観測性の**役割の違い**を示す。  
「高機能な監視」ではないと明確にする。

**■ Illustrated 解説の共通ルール（Must）**  
- 図は「答え」ではなく「**問いの可視化**」として使う。図を見せた後、必ず「この図は何を前提にしているか」を問い返す。
- 図に描かれた要素（矢印・分割線・箱）が、それぞれ**どの責任を引き受けているか**を言葉にする。
- 図を自作・改変する場合（How）も、上記の**思想の核（Must）は変えない**。見せ方は自由、伝える本質は共通。

## 7. 誤解しやすいポイント（全体）

各回の Must にも記載していますが、  
勉強会全体で特に繰り返し起きやすい誤解を、Lecturer が先回りで解くために整理します。  

教科書の重点思想と対応しています。

- **「なぜ生まれたか」を飛ばして「使い方」から入ってしまう。**   
→ すべての技術は、まず「**なぜその技術が生まれたのか / 何を解決するための技術なのか**」から入る。  
使い方は後。
- **技術を単独で教えてしまう。**   
→ コンテナ・Kubernetes・GitOps・可観測性は独立した知識ではなく、  
**技術同士の関係性**の中で意味を持つ。前後の章とつなぐ。
- **Kubernetes を魔法のツールとして語ってしまう。**  
→ **Kubernetes は魔法のツールではない。** あるべき状態を保つために差分を埋め続けるだけで、  
資源管理そのものは OS に委ねている。万能ではない。
- **マイクロサービスを「進んだ正しい形」として教えてしまう。**   
→ **マイクロサービスは万能ではない。** 設計上の選択肢であり、複雑さは消えず**置き場所が変わる**だけ。
- **クラウドを絶対視してしまう。**  
→ **クラウドは絶対的な正解ではない。** オンプレやハイブリッドにも活かせる場面があり、  
判断の軸（何を成立させ、どこまで責任を持つか）は環境が変わっても変わらない。
- **宣言的制御を「自動化」と混同する。**  
→ 宣言的制御は「どうするか」ではなく「**どうなっていてほしいか**」を定義し、  
差分を埋め続けること。単なる自動実行ではない。
- **可観測性を「たくさんログを取ること」と誤解する。**  
→ 目的は情報を増やすことではなく、**観測情報から状態を判断できる**こと。  
「どこまで観測するか」を決める設計。
- **GitOps を「自動デプロイツール」と誤解する。**  
→ 本質は再現性と、**判断を構造として固定し、戻せる**こと。ツール（ArgoCD/Flux）は手段。

## 9. 講義資料作成時の注意事項

最後に、資料を作る際の運用上の注意点をまとめます。

### 9.1 構成の型を守る

各回の資料は、**「前提の引き継ぎ → 背景（なぜ生まれたか）→ 構造の説明 → 誤解の解消 → 次への問い**」  
という流れを基本形にしてください。  

この型が、教科書の「往復して積み上げる」構造と一致します。  
いきなり操作やツール紹介から始めないでください。

### 9.2 Must は削らない、How は自由に足す

時間が足りないときに削るのは **How（デモ・事例・深掘り**）であり、Must（背景・必須メッセージ・誤解の解消・図の観点）ではありません。

**講義時間は1回あたり**最大120分を原則**とし、状況に応じて配分を変えて構いませんが、  
**終了時間が早まることはあっても遅くなることはないように**設計してください。

### 9.3 用語と正式名称

本ガイドラインおよびプロジェクトでは、講師を **Lecturer**、図版を **Illustrated** と表記します。  
資料内の表記もこれに揃えてください。  

なお、**英語版資料は参照しません**（日本語版が正式版）。英語版との整合性確認は不要です。

### 9.4 ツール名は「手段」として扱う

ArgoCD / Flux / Istio / Linkerd / Prometheus / Grafana / Jaeger / OpenTelemetry などの製品名は、  
**構造を成立させる手段の例**として紹介してください。  

「このツールを使えば解決する」という語り方は避け、  
「**この構造を成立させるための一つの実装**」として位置づけます。

### 9.5 断定を避け、判断を促す

クラウドネイティブに絶対の正解はありません。  

「こうすべき」と断定するより、「**前提が変わればこう変わる**」「あなたなら何を引き受けるか」  と、  
受講者自身の判断を促す語り口を基本にしてください。

### 9.6 資料はレビューを経る

作成した資料は、必要に応じてプロジェクト側のレビュー（教材意図・設計思想との整合確認）を受けてください。  
Lecturer 個人の解釈が、教科書の Must と食い違っていないかを確認する工程です。

### 9.7 教科書のアップデートへの参加

Lecturer は、講義を通じて気づいた「伝わりにくい点」「誤解されやすい点」を、  
教科書本体の改善提案としてプロジェクトへフィードバックしてください。  
ガイドライン自体も、運用を通じて更新していく前提の文書です。
