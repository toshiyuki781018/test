# 第2章：継続的デリバリーと GitOps

## ■この章で扱うこと

前章では、アプリケーションをどのような単位で分けるか  
という設計上の判断を扱いました。

しかし、アプリケーションを分けただけでは、  
システムは運用できません。

分割されたアプリケーションは、

- 繰り返し変更され
- 更新され
- 環境に反映され続ける

必要があります。

マイクロサービス化によって  
アプリケーションの変更は複数のサービスに分散します。

その結果、システムの中には  
**変更という物流** が生まれます。

この章では、この **「変更を届け続ける」という物流** を  
どのように管理するのかを整理します。

ここでの目的は、特定のツールを使えるようになることではありません。

なぜ、現在のクラウドネイティブな環境では  
このようなデリバリーの形が前提になっているのかを理解することが  
この章のゴールです。

---

## 1. 「一度デプロイすれば終わり」ではなくなった理由

従来のシステムでは、デプロイは **イベント** でした。

- リリース日を決め
- 手順を確認し
- 問題が起きないことを祈りながら反映する

このやり方は、

- 変更頻度が低く
- 影響範囲も限定的

な環境では成立していました。

しかし、アプリケーションが分割され、  
変更の単位が小さくなると、この前提は崩れます。

- 小さな変更が頻繁に発生する  
- すべてを人手で管理できなくなる  
- 「誰が、いつ、何を変えたか」が重要になる  

ここから、**変更を前提とした運用** が必要になります。

---

## 2. 継続的デリバリーとは「自動化」ではない

継続的デリバリーという言葉は、しばしば  
「自動デプロイ」と同一視されます。

しかし、本質はそこではありません。

継続的デリバリーが目指しているのは、

**変更を、いつでも安全に環境へ反映できる状態を保つこと**

です。

そのために必要なのは、

- 変更内容が明確であること  
- 変更手順が再現可能であること  
- 問題が起きたときに元に戻せること  

つまり、

**人が頑張らなくても成立する状態**

を作ることです。

---

## 3. なぜ「人が直接デプロイしない」構造になるのか

クラウドネイティブな環境では、  
デプロイを人の操作に依存させない構造が一般的になっています。

これは、人が信用できないからではありません。

理由はもっと単純で、

- 人の操作は記録が残りにくい  
- 手順が微妙に変わりやすい  
- 再現性を保つのが難しい  

からです。

変更が頻繁に発生する状況では、

**「誰がやったか」より「何が行われたか」**

が重要になります。

そのため、デプロイという行為そのものを  
**人の手から切り離す設計** が選ばれます。

---

## 4. GitOpsという考え方

GitOps は、この課題に対する一つの整理の仕方です。

GitOps では、

- 環境のあるべき状態を Git に定義し  
- 実際の環境を、その状態に近づけ続ける  

という構造を取ります。

Git が選ばれている理由は、

- 変更履歴が残る  
- 差分が明確になる  
- レビューや承認ができる  

といった特性が、**変更という物流を管理すること** に向いているからです。

GitOps は、ツールや製品の名前ではなく、  
**変更物流を管理するための考え方** だと捉えると理解しやすくなります。

---

## 5. Kubernetes と GitOps の関係

Kubernetes は、  
**あるべき状態を宣言的に定義できる基盤** です。

- 望ましい状態を記述する  
- 実際の状態との差分を調整する  

この特性は、GitOps の考え方と非常に相性が良いものです。

そのため

Git に定義された状態  
→ Kubernetes の状態を同期する

という構造が自然に成立します。

---

## 6. 継続的デリバリーは「判断を減らす仕組み」である

継続的デリバリーや GitOps は、  
人の判断を増やすための仕組みではありません。

むしろ逆で、

**人が毎回判断しなくても済むようにするための仕組み**

です。

- 何を変更するか  
- どの状態が正しいか  
- いつ反映されるか  

これらを構造として固定することで  
運用時の迷いを減らします。

---

## 7. GitOpsという構造を成立させる仕組み

GitOps の構造は、

- Git を監視する  
- クラスタの状態を確認する  
- 差分があれば同期する  

という処理を継続的に回すことで成立します。

この役割を担うものが  
**GitOpsツール** と呼ばれるものです。

---

## 8. GitOpsツールという実装例

GitOps の代表的な実装として

- Argo CD  
- Flux  

があります。

どちらも

Git の状態  
クラスタの状態

を比較し、

差分があれば同期する

という GitOps の構造を Kubernetes 上で実装しています。

重要なのは、

**ツールの違いではなく構造を理解すること**

です。

---

## ■通信というもう一つの物流

ここまで見てきたのは、  
**変更という物流** です。

しかし、マイクロサービス化されたシステムでは  
もう一つの物流が存在します。

それは

**サービス間通信**

です。

- サービス同士が呼び出し合う  
- 通信が増える  
- 通信の状態を管理する必要がある  

つまり、

マイクロサービス化された環境では

**変更の物流**  
**通信の物流**

という二つの物流が存在します。

この章では、変更の物流を扱いました。

次の章では、もう一つの物流である

**サービス間通信**

をどのように管理するのかを整理します。

---

## ■この章のまとめ

この章では、

- 変更が前提になった環境では  
- 人の操作に依存しない形で  
- 状態を管理する必要がある  

という点を整理しました。

継続的デリバリーや GitOps は  
**変更という物流を安全に管理する仕組み** です。

しかし、マイクロサービス環境では  
変更だけではなく

**通信という物流**

も存在します。

次の章では、この通信を  
**サービス間通信制御** という観点から扱います。
