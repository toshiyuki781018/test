# 第3部：アプリケーションを届け・つなぐ技術

マイクロサービスは、単なる技術の選択ではありません。

システムの規模が大きくなり、チームが増え、変更頻度が高くなると、  
単一のアプリケーションでは変更や運用を管理することが難しくなります。

マイクロサービスは、  
こうした **組織と変更速度のスケールに対応するための構造** として生まれたものです。

---

## ■この部で扱うこと

マイクロサービス環境では、システムの中に二つの **物流（流れ）** が生まれます。

一つは **変更という物流（変更の流れ）** です。  
アプリケーションの更新を、どのように安全に届け続けるのか。

もう一つは **通信という物流（通信の流れ）** です。  
分割されたサービス同士が、どのようにつながり、どのようにやり取りを行うのか。

この第3部では、マイクロサービス環境において  
アプリケーションを **届け、つなぐための構造** を整理します。

ここで扱う内容は、特定のツールを覚えることではありません。
マイクロサービス環境で発生する複雑さを **どこで引き受ける設計なのかを理解すること**  それが、この部の目的です。

---

## ■この部の構成

この部では、次の三つの観点からマイクロサービス環境を整理します。

### 1. マイクロサービスとアプリケーション構成

まず、アプリケーションをどのような単位で分割し、  
どのような構成で動かすのかを整理します。

ここでは、

- UI / API / DB の役割
- Kubernetes の基本リソース（Pod / Service / Ingress）

といった観点から、**クラウドネイティブアプリケーションの構造**を理解します。


### 2. 変更を届ける仕組み  
— 継続的デリバリーと GitOps —

マイクロサービス化された環境では、アプリケーションの変更が頻繁に発生します。

そのため、

- 変更をどのように管理するのか  
- 環境へどのように反映するのか  

という **変更という物流（変更の流れ）** を仕組みとして扱う必要があります。
ここでは、継続的デリバリーや GitOps の考え方を通して、**変更を安全に届け続ける構造**を整理します。


### 3. サービス間通信制御  
— Istio / Linkerd —

アプリケーションを分割すると、サービス同士はネットワーク越しに通信するようになります。

ここでは、

- 通信の失敗をどう扱うのか  
- 通信の振る舞いをどこで制御するのか  

といった **通信という物流（通信の流れ）** をどのように設計として扱うのかを整理します。

その一つの実装として、サービスメッシュ（Istio / Linkerd）という考え方を扱います。

---

## ■この部で大切にしたいこと

この部で扱うのは、特定のツールの使い方ではありません。

大切なのは、

- どの技術を選ぶか
- どのツールを導入するか

という判断の前に、  
**その複雑さをどこで引き受ける設計なのか**  
を理解することです。

クラウドネイティブ環境では、アプリケーションを分割することで

- 変更
- 通信

という二つの物流が生まれます。

この部では、それらをどのように扱うのかを  
構造として整理していきます。
