# 第3部：応用ステップ  〜 アプリケーションを届け、つなぐ 〜

## ■この部の目的
（副題案：設計判断としてのクラウドネイティブアプリケーション）

第1部では、クラウドネイティブを成り立たせている  
**考え方・前提条件・構造** を整理してきました。

第2部では、それらが Kubernetes 上でどのような挙動として現れるのかを  
実際に手を動かしながら体感しました。

第3部では、それらを前提として、アプリケーションを  
- どのように分け
- どのように届け
- どのようにつなぐか

という **設計上の判断** を扱います。

ここでのゴールは、特定の構成や技術を「正解」として
覚えることではありません。

アプリケーションを構成するときに

- 何を前提にしているのか
- どの制約を受け入れているのか
- どこから先を仕組みに任せ、どこを自分たちで引き受けるのか

といった点を、**意図を持って説明できる状態になること** が、この部の目的です。

言い換えると、第3部では
**マイクロサービス化によって増える「変更」と「通信」という物流を、どのように管理するか**
という設計上の判断を整理していきます。

## ■第3部で大切にしたいこと

この部では、次の点を意識して進めます。

- 「流行している構成」より「なぜそう分けたか」
- 「便利な仕組み」より「誰が複雑さを引き受けているか」
- 「できる／できない」より「どういう前提なら成立するか」

クラウドネイティブな設計には、唯一の正解は存在しません。  
第3部では判断を固定するのではなく、**判断できる視点を増やすこと** を重視します。

## ■クラウドネイティブな構成が必要になる理由

システムが小さいうちは、
アプリケーションを一つの構成として管理していても、大きな問題は起きません。

しかし、サービスの数が増え、機能が増え、
複数のチームが同時に開発を進めるようになると、

- 変更の回数が増える
- サービス同士の通信が増える

といった形で、システムの中を流れる **「変更」と「通信」** が急激に増えていきます。

このようにシステムの規模が大きくなると、  
アプリケーションの中で扱う **物流（変更と通信）** が増え、その管理が難しくなります。

クラウドネイティブで使われる多くの仕組みは、  
この増え続ける物流をどのように管理するかという問題から生まれてきました。

たとえば

- 変更を安全に届け続けるための仕組み
- サービス同士の通信を制御する仕組み

といった考え方は、  
すべてこの **「物流の増加」** に対応するためのものです。

第3部では、こうした背景を踏まえながら、  
アプリケーションを どのように分け、どのように届け、どのようにつなぐのか  
という設計上の判断を整理していきます。

## ■なぜ、いきなり構成例から始めないのか

「クラウドネイティブアプリケーション」と聞くと、

- マイクロサービス構成
- GitOps によるデプロイ
- サービスメッシュの導入

といった具体的な構成を思い浮かべるかもしれません。

しかし、これらを前提や理由を整理しないまま見てしまうと、

- なぜ分けているのか分からない
- なぜこの仕組みが必要なのか腑に落ちない
- 自分たちの状況に当てはまるか判断できない

といった状態になりがちです。

そのため第3部では、構成例や技術の説明に入る前に、  
**「どう考えて構成を選ぶのか」** から整理 します。

## ■第3部の進め方

この部は、大きく次の流れで進みます。

#### 1. アプリケーションを、どの単位に分けて考えるか

（第1章：マイクロサービスと構成要素）

- マイクロサービスを「目的」ではなく設計上の選択肢として扱う
- UI / API / データといった構成要素を、責務の観点で整理する
- 分割によって何が楽になり、何が難しくなるのかを確認する

ここでの目的は、「どう分けるべきか」を決めることではありません。  
**分け方を判断するための視点を揃えること** が、この章の役割です。

#### 2. 分けたアプリケーションを、どう届け続けるか

（第2章：継続的デリバリーと GitOps）

- なぜ人が直接デプロイしない構造になっているのか
- なぜ状態管理を Git に寄せる考え方が出てきたのか
- 継続的に変更を届けるための **前提条件** を整理する

ここでは、判断よりも 前提の理解 を重視します。

#### 3. 分けたサービス同士を、どうつなぎ、どう制御するか

（第3章：サービス間通信制御）

- サービス間通信が複雑になる理由
- 通信を「発生するもの」ではなく「制御対象」として捉える視点
- その複雑さを、どこまで仕組みに任せるかという判断

ここでの目的は、特定の技術を採用することではなく、  
**通信に関する責任の所在を意識できるようになること** です。

## ■第3部 第1章へ

それではまず、**アプリケーションをどの単位で捉えるか**  
という話から始めましょう。

次の章では、マイクロサービスという言葉に振り回されずに、

- なぜ分割という発想が出てくるのか
- 分割はどのような前提の上で成立するのか

を整理していきます。

ここでの目的は、「マイクロサービスを採用するかどうか」を決めることではありません。  
**設計判断として分け方を考えられる状態になること** が、この章の役割です。
